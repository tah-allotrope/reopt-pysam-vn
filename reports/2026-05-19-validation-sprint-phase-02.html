03 / Math / Algorithm Used

Math / Algorithm Used

No novel math was implemented or needed for this phase. The API solve pipeline is a REST submission + polling workflow:

  1. Submission: POST scenario JSON to /api/reopt/stable/job?api_key=<key>
  2. Polling: GET /api/reopt/stable/job/<run_uuid>?api_key=<key> every 5s until status is "completed" or "error"
  3. Normalization: API v3 wraps results in an outputs key; flatten to match local Julia solve format for downstream modules

The optimization itself is handled server-side by REopt v3 (HiGHS solver) — not computed locally.

06 / Limitations / Second-Best Alternative

Limitations / Second-Best Alternative

  • Scope: Only 2 of 6 TOU scenarios were solved (saigon18 pair). Ninhsim and north_thuan remain unsolved — this is within scope per sprint plan defaults.
  • API v3 schema mismatch: The API returns results wrapped in an outputs key, different from the flat format of local Julia solves. The normalization fix was applied but not yet tested against all downstream modules.
  • Network dependency: API solves depend on network connectivity to developer.nlr.gov. Domain is confirmed working (3 months before old nrel.gov expiration).
  • Rate limits: 1,000 req/hr standard quota. Two solves consumed ~6 requests (2 POST + 6 poll). Not a concern for this sprint.
07 / Errors / Warnings / Flags

Errors / Warnings / Flags Encountered

  • API v3 outputs wrapper: The initial API response had results nested under outputs key rather than at the top level. Downstream modules (equity_irr.py, tou_financial_delta.py) expected flat format. Fixed by adding normalization to solve_via_api.py.
  • Financial delta CSV: Some expected keys (year_one_electric_cost, lifecycle_electric_cost) are named differently in API v3 vs local Julia output. The CSV correctly shows NPV but not energy cost columns for the API-solved scenarios — this is a data contract difference, not a defect.
  • No errors encountered: Both API submissions completed with status=optimal. No timeouts, no schema validation errors, no rate limit issues.
