03 / Math / Algorithm Used

Math / Algorithm Used

Deterministic Scenario Hashing

Each materialized scenario is hashed to enable cache-first execution. The hash input is a composite payload containing the full resolved scenario dict, regime_id, assumption_set_id, and registry version. The payload is canonicalized via json.dumps(payload, sort_keys=True, separators=(",",":"), ensure_ascii=True) to ensure key ordering and whitespace are deterministic across runs and platforms. The canonical string is then fed to SHA-256, truncated to 16 hex characters.

hash = sha256(canonical_json)[:16]

Deep-Merge Override Resolution

Assumption sets apply overrides onto the base scenario using recursive deep-merge: for each key in the override, if both the base and override values are dicts, recurse; otherwise the override value replaces the base. This preserves nested scenario structure (e.g. ElectricTariff.monthly_demand_rates) without requiring assumption sets to duplicate entire top-level blocks.

Cache-First Matrix Execution

Before invoking the Julia solver, build_regime_matrix() checks whether a run_manifest.json with a successful terminal status (scenario_built_no_solve, optimal, or cached) already exists at the deterministic path. If so, the run is marked cached and skipped unless --force is set. This prevents redundant 2+ minute Julia compilations on repeated runs.

06 / Limitations / Second-Best Alternative

Limitations / Second-Best Alternative

  • Julia compilation overhead — Each subprocess call compiles the full project (~2–3 min). A 5-regime × 3-assumption matrix takes ~30 min. The second-best alternative would be a persistent Julia server (DaemonMode.jl or PackageCompiler sysimage), but that introduces deployment complexity beyond Phase 3 scope.
  • No parallel execution — Matrix runs are sequential. Julia memory footprint (~800 MB) allows parallel processes on modern machines, but the current subprocess.run() loop is serial. A future enhancement could use concurrent.futures with a configurable concurrency cap.
  • Assumption sets are metadata-only for some regimes — The repriced_multipliers and capacity_payment_preview sets carry marker metadata but no scenario-level overrides yet, because the real multiplier data (Decision 963 repriced) and BESS capacity payment tariff have not been published by MOIT.
  • No Julia-side regime awareness — The Julia runner receives a fully resolved scenario JSON and a result directory. It has no concept of regime_id or assumption_set_id. This is by design (keep REopt.jl unmodified), but means the Julia side cannot validate regime coherence independently.
07 / Errors / Warnings / Flags

Errors / Warnings / Flags Encountered

  • tilt_rule key skip — Both Python and Julia tech-cost defaults contained a tilt_rule string key that was being set as a numeric default on PV dicts, causing type errors in downstream consumers. Fixed by adding if k == "tilt_rule": continue in both languages (mirrored parity fix).
  • _meta propagation gap — The regime registry loader was discarding the top-level _meta block (containing registry_version) when extracting regimes_raw["data"]. Fixed by merging _meta into the regimes payload in both Python and Julia loaders so resolve_vietnam_regime() can propagate the version into run manifests.
  • Expected warning on draft export regime — test_regime_specific_export_fraction emits a UserWarning that the 50% export cap is stored but not enforced by plain REopt.run_reopt(). This is documented and expected behavior (post-processing only).
